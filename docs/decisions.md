## 2026-03-12 | Negative 데이터 최종 기준을 `AI Hub + MUSAN + FSD50K`로 확정

### 결정

- `Common Voice KO`는 최종 기준에서 제외한다.
- negative 데이터는 `AI Hub 자유대화 음성(일반남여) + MUSAN + FSD50K` 조합으로 간다.

### 이유

- AI Hub가 한국어 일상 발화 품질 측면에서 더 적합하다.
- MUSAN/FSD50K는 환경 잡음 다양성 보강에 유리하다.
- 실제 작업 방향도 이미 이 조합으로 이동하고 있었다.

## 2026-03-12 | Negative 데이터 공통 출력 스펙을 먼저 고정

### 결정

- AI Hub, MUSAN, FSD50K의 최종 출력 형식을 공통 스펙으로 맞춘다.
- 공통 스펙은 `16kHz / mono / WAV(PCM_16) / 3.0초 고정 클립`이다.
- 전처리는 `MUSAN`부터 먼저 시작한다.

### 이유

- 세 데이터셋이 같은 포맷이어야 이후 feature extraction과 학습 코드가 단순해진다.
- AI Hub 다운로드를 기다리는 동안 MUSAN부터 처리해 전체 일정을 앞당길 수 있다.

## 2026-03-12 | Positive 증강 산출물을 `clean`과 `mixed` 계층으로 분리

### 결정

- `positive_aug`를 `clean`, `mixed_noise`, `mixed_speech`, `_manifests` 구조로 나눈다.
- 다만 mixed background 증강 실행은 negative 데이터셋 준비가 모두 끝난 뒤에 시작한다.

### 이유

- clean 증강과 background mixing 증강을 분리해야 비교와 비율 조절이 쉽다.
- 현재 우선순위는 negative 데이터셋 완성이므로 mixed 증강은 뒤로 미룬다.

## 2026-03-12 | 대용량 다운로드 파일은 즉시 삭제하지 않는 원칙 채택

### 결정

- 손상 의심이 있는 대용량 다운로드 파일도 즉시 삭제하지 않는다.
- 먼저 무결성 확인, 헤더 점검, 이어받기 가능 여부를 확인한다.
- 큰 압축 병합/해제는 조용한 모드 또는 로그 파일 기반으로 수행한다.

### 이유

- 일부라도 다운로드된 대용량 파일을 성급히 삭제하면 시간 손실이 크다.
- 압축/네트워크 문제는 재검증이나 이어받기로 해결 가능한 경우가 있다.

## 2026-03-13 | Mixed positive 증강은 원본 기반으로 생성

### 결정

- `mixed_noise`, `mixed_speech`는 기존 clean 증강본이 아니라 원본 positive에서 직접 생성한다.
- 시작 비율은 전체 원본 positive의 50%를 mixed 대상으로 사용한다.

### 이유

- 이미 증강된 오디오에 다시 배경 혼합을 넣으면 왜곡이 누적되기 쉽다.
- clean 증강과 mixed 증강을 원본에서 독립적으로 만드는 편이 분포 관리와 비교가 쉽다.

## 2026-03-13 | 긴 feature 추출 작업은 부분 재실행과 주기 로그를 지원

### 결정

- `04_extract_features.py`는 특정 split만 다시 실행할 수 있도록 `--groups` 옵션을 둔다.
- 긴 CPU 작업에서는 `--progress-interval-sec`로 주기적인 별도 진행 로그를 출력한다.

### 이유

- 인터럽트가 나도 전체를 처음부터 다시 돌리지 않기 위해서다.
- 장시간 작업은 터미널 외부 진행률 확인이 어려우므로, 일정 주기 로그가 필요하다.

## 2026-03-13 | 1분 이상 실행 코드는 1분 주기 로그를 기본 정책으로 채택

### 결정

- 1분 이상 걸릴 가능성이 있는 모든 실행 코드는 기본적으로 1분마다 진행 로그를 출력한다.
- 가능한 경우 부분 재실행 옵션도 함께 제공한다.

### 이유

- 사용자가 별도로 질의하지 않아도 터미널 출력만으로 상태를 파악할 수 있어야 한다.
- 긴 데이터 처리와 학습 작업에서 인터럽트 이후 복구 비용을 줄이기 위해서다.

## 2026-03-13 | 05_train.py는 openWakeWord 구조를 따르되 로컬 독립 구현으로 유지

### 결정

- `05_train.py`는 openWakeWord의 FC classifier 구조를 따른다.
- 다만 `openwakeword.train.Model`를 직접 import하지 않고, 동일한 모델 구조를 로컬 스크립트에 구현해 사용한다.

### 이유

- 현재 환경에서 `openwakeword.train` import 시 `torchinfo` 같은 부가 의존성이 걸려 실행 안정성이 떨어진다.
- 모델 구조는 유지하면서 실행 의존성을 줄이는 편이 지금 단계에서 더 실용적이다.

## 2026-03-13 | baseline과 개선 실험은 run 단위 아카이브로 관리

### 결정

- `05_train.py`의 학습 결과는 `wake_word/models/hi_popo/runs/<run_name>/` 아래에 저장한다.
- 각 run마다 체크포인트, 학습 히스토리, 메타데이터를 함께 남긴다.
- 최신 결과는 별도로 `wake_word/models/hi_popo/` 루트에도 갱신한다.

### 이유

- baseline 학습과 이후 성능 개선 실험을 명확히 분리해 비교하기 위해서다.
- 단순 문서 로그를 넘어서, 실제 재현 가능한 산출물과 설정을 같이 보관해야 하기 때문이다.

## 2026-03-13 | baseline 평가는 작은 샘플 확인 후 medium 규모로 확장

### 결정

- baseline은 처음부터 전체 데이터를 오래 학습하지 않고, 작은 샘플 baseline으로 수렴 여부를 먼저 확인한다.
- 수렴이 확인되면 `baseline_medium`처럼 positive 전체와 확대된 negative subset으로 다시 학습한다.

### 이유

- 빠르게 학습 가능성과 metric 방향성을 확인한 뒤, 더 큰 데이터로 안정적인 baseline을 잡는 편이 효율적이다.
- 현재 `baseline_medium` 결과는 recall은 충분히 높지만 false positive rate는 아직 추가 개선 여지가 있다.

## 2026-03-13 | 최종 학습 전에는 좁은 grid search로 핵심 하이퍼파라미터를 먼저 고른다

### 결정

- 최종 전체 데이터 학습 전에 baseline 구조에서 핵심 하이퍼파라미터만 좁게 탐색한다.
- 탐색 축은 우선 `lr`, `negative_weight`, `layer_dim`, `n_blocks`로 제한한다.
- 구조 변경이나 새로운 손실함수 도입은 이번 사이클에서는 하지 않는다.

### 이유

- 한 번만 더 크게 돌릴 계획이라면, 전체 데이터 학습 전에 저위험 파라미터 최적화를 먼저 하는 편이 낫다.
- A100 환경에서는 이 정도 탐색 비용이 감당 가능하고, baseline 대비 개선 가능성이 높다.

## 2026-03-13 | 최종 전체 학습은 grid search best trial 파라미터를 사용

### 결정

- 최종 전체 데이터 학습은 `baseline_grid_v1_trial40`의 파라미터를 사용한다.
- 사용 파라미터는 아래와 같다.
  - `lr=0.0005`
  - `negative_weight=5.0`
  - `layer_dim=64`
  - `n_blocks=2`

### 이유

- 동일한 validation 조건에서 `baseline_grid_v1_trial40`이 가장 좋은 종합 score를 기록했다.
- baseline 대비 false positive rate가 크게 줄었고, recall도 유지됐다.
- 구조 변경 없이 저위험 파라미터 최적화만 반영하는 보수적 경로로 적합했다.

## 2026-03-13 | 장기 목표는 Jetson 내부 재사용형 음성 에이전트 SDK

### 결정

- 이 프로젝트의 장기 목표는 단순한 wake word 데모가 아니라, Jetson 내부에서 재사용 가능한 음성 에이전트 SDK/플랫폼을 만드는 것이다.

- 안면 인식, 표정 인식, 특정 위치 이동, 순찰 모드 같은 다른 로봇 기능과의 연결이 상위 목적에 더 가깝다.

## 2026-03-13 | 자동 push 운영에서도 `secrets/`는 절대 추적하지 않는다

### 결정

- 자동 push 운영과 무관하게 `secrets/` 아래 파일은 Git 추적 및 원격 반영 대상에서 완전히 제외한다.
- 커밋 및 push 전에 `secrets/`가 tracked 또는 staged 상태가 아닌지 반드시 확인한다.

### 이유

- 자동 push는 편의성을 높이지만, 민감 정보 경계가 흐려지면 바로 사고로 이어진다.
- `secrets/`는 공개 문서와 완전히 분리된 로컬 전용 영역으로 유지해야 한다.

## 2026-03-16 | 초기 구현은 단순한 주경로 우선, 오버엔지니어링 금지

### 결정

- 필요한 기능은 확실하게 구현하되, 처음부터 과도한 예외처리나 추측성 fallback을 넣지 않는다.
- 실제로 발생한 에러를 보고 필요한 만큼만 예외처리와 보호 로직을 보강한다.
- 과도한 추상화, 아직 쓰지 않는 옵션, 조기 범용화는 기본적으로 하지 않는다.

### 이유

- 처음부터 모든 경우를 상정해 방어 코드를 늘리면 구현 속도와 가독성이 같이 떨어진다.
- 실제로 발생하지 않은 문제를 미리 막으려는 코드보다, 실제 실패를 보고 보강한 코드가 더 정확하고 유지보수하기 쉽다.
- 현재 프로젝트 단계에서는 빠르게 동작하는 단순한 구현을 먼저 확보하는 편이 더 실용적이다.

## 2026-03-16 | 함수 시그니처 타입 힌트는 기본적으로 쓰지 않는다

### 결정

- 함수 정의에는 반환 타입과 세부 타입 힌트를 기본적으로 넣지 않는다.
- 예를 들어 `-> list[str]`, `arg: dict[str, float]` 같은 표기는 현재 프로젝트 기본 스타일에서 제외한다.

### 이유

- 현재 단계에서는 타입 정보를 엄격히 적는 것보다, 코드를 빠르게 읽고 바로 수정할 수 있는 단순한 시그니처가 더 중요하다.
- 실제로 사용하지 않는 타입 표현이 늘어나면 코드가 길어지고, 사용자 선호와도 맞지 않는다.
- 필요한 경우는 나중에 실제 요구가 생겼을 때 그 지점만 보강하면 된다.

## 2026-03-16 | 모든 함수는 한국어 docstring을 기본으로 작성

### 결정

- 모든 `def`, `async def` 바로 아래에는 한국어 docstring을 넣는다.
- docstring 형식은 최소한 `기능`, `입력`, `반환` 3개 항목을 포함한다.

### 이유

- 현재 프로젝트는 세션을 오가며 이어서 작업하는 일이 많아, 함수 단위 설명이 바로 보이는 편이 맥락 복구에 유리하다.
- 코드 작성자 본인도 시간이 지나면 세부 맥락을 잊기 쉬우므로, 함수 아래 설명을 기본 습관으로 두는 편이 유지보수에 유리하다.
