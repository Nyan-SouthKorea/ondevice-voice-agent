## 1. 프로젝트 개요

케어 로봇에 탑재되는 음성 명령 시스템. 사용자가 "하이 포포"라고 부르면 로봇이 깨어나 음성으로 대화하는 시스템을 개발한다.

**현재 단계**: PoC 데모 준비 중

---

## 2. 하드웨어 환경

### 추론 환경 (Jetson)
- **Jetson** = Jetson Orin Nano Developer Kit 8GB
- SD카드에 OS 설치 (Ubuntu 22.04)
- ROS2 Humble 풀 설치
- onnxruntime-gpu 1.23.0 + TensorRT 10.3.0
- 이 문서에서 "Jetson"은 항상 이 하드웨어를 의미함

### 학습 환경 (Linux PC)
- **GPU**: NVIDIA A100 80GB PCIe
- **OS**: Ubuntu (Linux)
- **CUDA**: 11.8 / cuDNN 8.9
- **Python**: 3.10
- **PyTorch**: 2.7.1+cu118
- **conda 환경명**: `wake_word`
- 자세한 세팅 방법은 `docs/envs/wake_word_env.md` 참고

## 3. 핵심 개발 원칙

### 3-1. ONNX 우선
- 이 환경에서 `onnxruntime-gpu 1.23.0`이 완벽하게 동작함
- **모든 개발물은 ONNX로 실행 가능한지 반드시 검토**
- 학습은 어디서든 하되, 추론은 항상 ONNX로
- TensorRT 10.3.0도 설치되어 있으므로 ONNX → TensorRT 변환도 가능

### 3-2. 온디바이스 추론
- Jetson Orin Nano 8GB RAM 제약을 항상 고려
- 모델 추천 시 ONNX Export 가능 여부, TensorRT 변환 가능 여부 함께 검토
- 무거운 모델보다 경량 모델 우선

### 3-3. 라이선스 관리
- 상업적 사용 가능 여부를 항상 확인 (Apache 2.0, MIT, CC0 우선)
- GPL 계열은 코드 공개 의무 발생하므로 주의
- CC BY-NC 계열은 PoC에서만 허용, 제품화 전 반드시 교체

### 3-4. 모듈 구조 원칙
- 각 요소기술은 독립 폴더로 관리
- 모듈 파일과 데모 파일을 분리 (예: `stt.py` / `stt_demo.py`)
- 온디바이스 / API 방식을 래퍼로 통일 (예: `STT(api=False)`)

### 3-5. 문서 동기화 원칙
- 작업 진행 중 상태가 바뀌면 관련 문서를 즉시 갱신한다
- 최신 상태는 `docs/status.md`에 유지한다
- 중요한 의사결정 변경은 `docs/decisions.md`에 누적 기록한다
- 실제 작업 진행 내역은 `docs/logbook.md`에 시간순으로 남긴다
- 코드, 데이터, 운영 기준이 바뀌었는데 문서가 뒤처지지 않도록 항상 함께 관리한다

### 3-5-1. Git 연동 시 버전 관리 원칙
- Git으로 작업을 관리하는 경우, 코드 변경과 문서 변경을 가능한 한 같은 흐름 안에서 함께 정리한다
- 기능 추가, 데이터 처리 기준 변경, 학습 방식 변경, 디렉토리 구조 변경이 발생하면 관련 문서와 Git 상태를 같이 점검한다
- 중간 산출물과 재생성 가능한 대용량 파일은 버전 관리 대상에서 분리하고, 추적 대상은 의도적으로 선택한다
- 커밋 전에는 현재 변경 사항이 문서 기준과 일치하는지 확인하고, 추적되면 안 되는 파일이 포함되지 않았는지 점검한다
- 작업 단위가 정리됐다고 판단되면 중간 커밋은 수시로 수행한다
- 현재 운영 기준에서는 작업 단위가 정리되면 원격 `push`도 함께 수행한다
- 여러 주 원격을 함께 운영하는 경우, 같은 커밋을 모든 주 원격에 같은 타이밍에 반영한다
- `secrets/` 아래 파일은 어떤 경우에도 Git 추적 대상에 올리지 않는다
- 커밋과 push 전에는 `secrets/`가 staged 또는 tracked 상태가 아닌지 반드시 확인한다
- `secrets/` 또는 민감한 운영 메모가 추적 상태로 보이면 커밋/푸시를 멈추고 먼저 추적 해제 또는 로컬 전용 문서로 이동한다

### 3-5-2. 민감 정보 문서화 원칙
- 내부 주소, 계정 식별자, SSH 경로, 비공개 운영 메모는 공개 문서에 직접 남기지 않는다
- 이런 정보가 꼭 필요하면 `secrets/` 아래의 로컬 전용 문서로 분리해 관리한다
- 공개 리포에는 일반화된 설명만 남기고, 실제 접속 정보나 운영 절차는 로컬 메모로 분리한다
- 문서 정리 과정에서 민감 정보가 발견되면 최신 공개 문서에서는 제거하고, 필요한 경우에만 로컬 전용 문서로 이동한다
- 자동 push 운영 중에도 위 원칙은 예외 없이 유지한다

### 3-6. 대용량 데이터 처리 원칙
- 대용량 다운로드 파일은 손상 여부가 의심되어도 즉시 삭제하지 않는다
- 먼저 파일 헤더, 압축 무결성, 이어받기 가능 여부를 확인한다
- 이미 일부라도 다운로드된 파일은 가능하면 보존하고 `resume` 또는 검증을 우선 시도한다
- 삭제가 필요할 때는 진행 손실과 대체 경로를 함께 검토한 뒤 수행한다
- 큰 압축 병합/해제 작업은 가능한 한 로그 파일 또는 조용한 모드로 실행한다

### 3-7. 장시간 실행 로그 원칙
- 1분 이상 걸릴 수 있는 모든 실행 코드는 기본적으로 주기 로그를 출력한다
- 기본 주기는 1분이며, 진행률, 처리량, 현재 단계, 예상 남은 시간을 가능한 범위에서 함께 출력한다
- 사용자가 별도로 물어보지 않아도 상태를 확인할 수 있도록 터미널 출력만으로 진행 상황을 파악 가능해야 한다
- 인터럽트 이후 재실행이 잦은 작업은 부분 재실행 옵션과 함께 설계하는 것을 우선한다

## 5. Wake Word 개발 방침

### Wake Word: "하이 포포"

### 학습 전략
- **모델**: openWakeWord (Apache 2.0)
- **Backbone**: Google Speech Embedding (Apache 2.0, frozen ONNX)
- **분류기**: FC 2레이어(32유닛) → binary cross-entropy, PyTorch 학습 → ONNX export
- **학습 환경**: Linux PC (A100 80GB, CUDA 11.8, conda 환경 `wake_word`)
- **추론 환경**: Jetson (onnxruntime-gpu)
- **참고 노트북**: openWakeWord 레포 `notebooks/training_models.ipynb` (한국어 미지원이라 커스텀 파이프라인 작성)

### 데이터 전략
- **Positive 목표**: 1,000~2,000개
  - Edge TTS 한국어 10개 목소리 × 속도/피치 변형
  - 실제 녹음: 팀원 8명 × 12~15회 (총 ~100개)
  - librosa 증강으로 3~5배 확장
- **Negative**: AI Hub 자유대화 음성(일반남여) + MUSAN + FSD50K
  - 한국어 일상 발화는 AI Hub를 우선 사용
  - 환경음/잡음 다양성은 MUSAN, FSD50K로 보강
  - 세부 수집 기준은 `docs/research/negative_datasets.md` 참고

### TTS 전략 (Positive 샘플 생성)
| 도구 | 라이선스 | 용도 |
|------|---------|------|
| Edge TTS | Microsoft 약관 | PoC 데이터 생성 (10개 목소리, 속도/피치 조절) |
| MeloTTS | MIT | 제품화 시 대체 (오프라인, 라이선스 안전) |

## 6. 개발 프로세스 (PDCA)

```
Plan  → 기술 조사, 모델 선정, 데이터 전략 수립
Do    → 학습 파이프라인 구현, 모델 학습, 모듈 코드 작성
Check → 데모 검증, 성능 지표 측정 (FAR/FRR), 엣지케이스 확인
Act   → 임계값 튜닝, 데이터 보강, 모델 교체, 다음 사이클 반복
```

각 사이클의 결과는 `docs/phases/` 에 기록한다.
