# Robot Voice Commander - 개발 방침

> 이 문서는 오래 유지할 원칙만 적는다. 현재 상태와 세부 진행 상황은 `docs/status.md`를 기준으로 관리한다.

## 1. 프로젝트 방향

- 목표는 Jetson 내부에서 재사용 가능한 음성 에이전트 SDK를 만드는 것이다.
- 현재 구현 순서는 `wake word -> VAD -> STT -> LLM -> TTS`다.
- 추론 타깃 하드웨어는 `Jetson Orin Nano Developer Kit 8GB`다.
- 학습/평가와 추론/배포는 같은 문서에 섞지 말고 역할별로 분리한다.

## 2. 핵심 개발 원칙

### 2-1. ONNX 우선

- 학습 환경과 별개로, 추론 경로는 항상 ONNX 기준 실행 가능성을 먼저 본다.
- Jetson에서 유지할 runtime 자산은 재현에 필요한 최소 집합만 둔다.

### 2-2. 온디바이스 우선

- 모델이나 라이브러리를 고를 때 Jetson RAM, 속도, 배포 단순성을 먼저 본다.
- API 경로는 빠른 통합용 보조 수단으로 쓸 수 있지만, 장기 기본값은 온디바이스 가능성을 검토한다.

### 2-3. 라이선스 관리

- 상업적 사용 가능 여부를 기본 검토 항목으로 둔다.
- GPL, CC BY-NC 같은 제약이 큰 선택지는 초기 검토 단계에서 바로 표시한다.

### 2-4. 단순한 모듈 인터페이스

- 각 요소기술은 독립 폴더와 공통 진입점으로 관리한다.
- backend가 여러 개여도 상위 사용법은 하나로 유지한다.
- 데모 파일과 실제 모듈 파일은 분리한다.

### 2-5. 오버엔지니어링 금지

- 처음부터 과도한 추상화, 미사용 옵션, 추측성 fallback을 넣지 않는다.
- 먼저 주경로를 단순하게 완성하고, 실제 문제를 본 뒤 필요한 만큼만 확장한다.

### 2-6. 코드 스타일 단순화

- 함수 시그니처는 현재 단계에서 지나치게 복잡하게 만들지 않는다.
- 모든 함수는 한국어 docstring으로 `기능 / 입력 / 반환`을 기본적으로 적는다.

### 2-7. 계획 기반 실행

- 작업은 시작 전에 짧은 계획을 세우고 순서대로 진행한다.
- 중간에 계획이 바뀌면 이유를 남긴다.

### 2-8. 장시간 실행 로그

- 1분 이상 걸릴 수 있는 실행은 진행 상태를 주기적으로 출력한다.
- 재실행이 비싼 작업은 가능하면 부분 재실행과 중간 상태 확인을 지원한다.

## 3. 문서 운영 원칙

### 3-1. 문서 역할은 하나씩만 가진다

| 문서 | 역할 |
|---|---|
| `README.md` | 레포 소개와 진입점 |
| `docs/README.md` | 문서 허브 |
| `docs/status.md` | 현재 상태와 다음 작업의 단일 기준 |
| `docs/decisions.md` | 현재 유효한 핵심 결정 |
| `docs/logbook.md` | 최근 작업 로그 |
| `docs/archive/` | 오래된 상세 기록 |
| `docs/research/` | 기술 조사와 선택 배경 |
| `docs/envs/` | 환경 세팅과 검증 절차 |
| `docs/reports/` | 사람이 읽는 결과 요약 |

- 같은 상태 설명을 여러 상위 요약 문서에 반복하지 않는다.
- 상태가 바뀌면 우선 `docs/status.md`를 갱신한다.
- 원칙이 바뀌었을 때만 `docs/개발방침.md`를 갱신한다.
- 새 에이전트나 새 세션의 시작 절차는 `docs/README.md`를 기준으로 통일한다.
- 에이전트에게는 원칙적으로 `docs/README.md`만 먼저 읽으라고 지시하고, 나머지는 그 문서가 안내하는 범위만 읽게 한다.

### 3-2. 상위 문서는 얇게 유지한다

- 루트 `README.md`에는 프로젝트 소개, 모듈 링크, 문서 진입점만 둔다.
- `docs/README.md`는 색인과 역할 설명만 둔다.
- `docs/project_overview.md`는 새 세션용 배경 설명 페이지로만 유지한다.

### 3-3. 결정과 로그는 분리한다

- `docs/decisions.md`에는 현재도 유효한 프로젝트 결정만 남긴다.
- 세부 과정, 시행착오, 실행 기록은 `docs/logbook.md` 또는 `docs/archive/`로 보낸다.
- 한 번 유효했던 결정이라도 지금 기준에서 더 이상 중요하지 않으면 archive로 내린다.

### 3-4. 최근 로그만 활성 문서에 둔다

- `docs/logbook.md`는 최근 세션만 유지한다.
- 문서가 길어지거나 현재 단계와 직접 관련 없는 항목이 많아지면 `docs/archive/`로 옮긴다.
- archive로 옮길 때는 활성 문서에 짧은 안내를 남긴다.

### 3-5. 생성 산출물과 사람이 유지하는 문서를 분리한다

- `stt/eval_results/**/*.md`, CSV, JSON 같은 자동 생성 결과물은 공식 문서가 아니다.
- 자동 생성 결과물은 모듈 출력 디렉토리에 두고, 사람이 계속 읽을 필요가 있는 경우에만 요약을 `docs/reports/`로 승격한다.
- 보고서에는 결론과 해석만 적고, 원본 수치는 산출물 경로로 연결한다.

### 3-6. 환경 문서는 별도 유지한다

- 설치 절차, venv 경로, 검증 명령은 `docs/envs/` 아래에서 관리한다.
- 상위 문서에는 환경 문서 링크만 두고 설치 명령을 반복하지 않는다.

### 3-7. 민감 정보는 공개 문서에 쓰지 않는다

- 실제 계정, 키, 내부 주소, 사용량 로그 같은 운영 정보는 `secrets/` 로컬 문서에서만 관리한다.
- 공개 레포 문서에는 일반화된 설명만 남긴다.

## 4. 자산과 데이터 운영 원칙

- 대용량 데이터셋, 중간 feature, 대형 학습 산출물은 기본적으로 리포에 포함하지 않는다.
- 다만 Jetson runtime에 직접 필요한 소형 자산은 예외적으로 포함할 수 있다.
- 문서용 스크린샷, GIF, 영상은 `docs/assets/` 아래에서 역할별로 정리한다.

## 5. 문서를 갱신할 때의 체크리스트

1. 이번 변경의 현재 상태 기준이 바뀌었는지 먼저 본다.
2. 바뀌었다면 `docs/status.md`를 우선 갱신한다.
3. 원칙이 바뀌었는지 보고, 바뀌었다면 `docs/개발방침.md`를 갱신한다.
4. 오래 유지할 핵심 결정이면 `docs/decisions.md`에 반영한다.
5. 최근 작업 흐름이 필요하면 `docs/logbook.md`에 짧게 남긴다.
6. 긴 상세 기록은 활성 문서에 쌓지 말고 `docs/archive/`로 보낸다.
