# Robot Voice Commander - 개발 방침

> 최초 작성: 2026-03-12
> 이 문서는 언제든지 자유롭게 편집 가능합니다.

---

## 1. 프로젝트 개요

케어 로봇에 탑재되는 음성 명령 시스템. 사용자가 "하이 포포"라고 부르면 로봇이 깨어나 음성으로 대화하는 시스템을 개발한다.

**현재 단계**: wake word + VAD 요소기술 완료, 실기 튜닝 및 통합 준비 단계

---

## 2. 하드웨어 환경

### 추론 환경 (Jetson)
- **Jetson** = Jetson Orin Nano Developer Kit 8GB
- SD카드에 OS 설치 (Ubuntu 22.04)
- ROS2 Humble 풀 설치
- onnxruntime-gpu 1.23.0 + TensorRT 10.3.0
- 이 문서에서 "Jetson"은 항상 이 하드웨어를 의미함
- Jetson runtime venv 세팅과 검증 절차는 `docs/envs/jetson_wake_word_env.md`에서 관리함

### 학습 환경 (Linux PC)
- **GPU**: NVIDIA A100 80GB PCIe
- **OS**: Ubuntu (Linux)
- **CUDA**: 11.8 / cuDNN 8.9
- **Python**: 3.10
- **PyTorch**: 2.7.1+cu118
- **conda 환경명**: `wake_word`
- 자세한 세팅 방법은 `docs/envs/wake_word_env.md` 참고

## 3. 핵심 개발 원칙

### 3-1. ONNX 우선
- 이 환경에서 `onnxruntime-gpu 1.23.0`이 완벽하게 동작함
- **모든 개발물은 ONNX로 실행 가능한지 반드시 검토**
- 학습은 어디서든 하되, 추론은 항상 ONNX로
- TensorRT 10.3.0도 설치되어 있으므로 ONNX → TensorRT 변환도 가능

### 3-2. 온디바이스 추론
- Jetson Orin Nano 8GB RAM 제약을 항상 고려
- 모델 추천 시 ONNX Export 가능 여부, TensorRT 변환 가능 여부 함께 검토
- 무거운 모델보다 경량 모델 우선

### 3-3. 라이선스 관리
- 상업적 사용 가능 여부를 항상 확인 (Apache 2.0, MIT, CC0 우선)
- GPL 계열은 코드 공개 의무 발생하므로 주의
- CC BY-NC 계열은 PoC에서만 허용, 제품화 전 반드시 교체

### 3-4. 모듈 구조 원칙
- 각 요소기술은 독립 폴더로 관리
- 모듈 파일과 데모 파일을 분리 (예: `stt.py` / `stt_demo.py`)
- 온디바이스 / API 방식을 래퍼로 통일 (예: `STT(api=False)`)

### 3-4-1. SDK 지향 아키텍처 원칙
- 최종 목표는 개별 모델 모음이 아니라, Jetson 내부에서 재사용 가능한 음성 에이전트 SDK를 만드는 것이다
- 각 요소기술은 다른 개발자가 쉽게 붙일 수 있도록 단순하고 일관된 인터페이스를 우선한다
- 프론트엔드, 백엔드, 로봇 기능 개발자가 음성 파이프라인을 서비스처럼 호출하거나 이벤트처럼 구독할 수 있는 구조를 지향한다
- wake word, VAD, STT, LLM, TTS는 서로 강하게 결합된 스크립트가 아니라, 교체 가능한 모듈로 설계한다
- 음성 파이프라인은 안면 인식, 표정 인식, 특정 위치 이동, 순찰 모드 같은 다른 기능과 쉽게 연결될 수 있어야 한다
- 즉시 구현 편의보다 이후 통합성과 재사용성을 우선한다

### 3-4-2. 오버엔지니어링 금지 원칙
- 필요한 기능은 확실하게 구현하되, 처음부터 과도한 추상화나 방어 로직을 넣지 않는다
- 실제로 쓰지 않은 예외처리, 추측성 fallback, 과도한 옵션화, 아직 필요하지 않은 범용화는 기본적으로 만들지 않는다
- 처음 구현은 주경로(happy path)를 단순하고 명확하게 완성하는 것을 우선한다
- 에러 상황은 실제로 발생한 뒤, 재현 가능한 입력과 맥락을 보고 필요한 만큼만 보강한다
- `try/except`로 문제를 미리 삼키기보다, 먼저 실패를 드러내고 원인을 확인한 뒤 수정하는 쪽을 선호한다
- 단, 데이터 손실 가능성, 보안 문제, 복구가 어려운 외부 부작용처럼 명백히 위험한 경우는 처음부터 최소한의 보호 장치를 둔다

### 3-4-3. 함수 시그니처 단순화 원칙
- 함수 정의에 반환 타입과 세부 타입 힌트(`-> list[str]`, `arg: dict[str, float]`)를 기본적으로 넣지 않는다
- 지금 단계에서는 코드 가독성과 수정 속도를 높이는 편이 더 중요하므로, 시그니처는 최대한 단순하게 유지한다
- 실제로 타입 정보가 꼭 필요하다고 확인된 경우가 아니면, 함수 인자와 반환값은 일반 Python 코드로만 표현한다

### 3-4-4. 함수 docstring 작성 원칙
- 모든 `def`, `async def` 바로 아래에는 한국어 docstring을 넣는다
- docstring은 최소한 `기능`, `입력`, `반환` 3개 항목을 포함한다
- 입력 인자가 없으면 `입력: 없음`, 반환값이 없으면 `반환: 없음`으로 명시한다
- 다른 사람이 코드를 처음 읽어도 함수 역할을 바로 이해할 수 있을 정도로 친절하게 쓴다

### 3-4-5. 계획 기반 실행 원칙
- 모든 작업은 시작 전에 `1. 2. 3.` 형태의 계획으로 먼저 정리한다
- 작업은 계획 순서대로 하나씩 이행한다
- 중간에 계획이 바뀌면 바뀐 이유와 수정된 순서를 다시 기록한다
- 작업을 마친 뒤에는 처음 세운 계획 대비 실제로 모두 끝났는지 반드시 다시 확인한다
- 최종 보고에는 결과만 적지 말고, 계획대로 이행됐는지 여부도 함께 점검한다

### 3-5. 문서 동기화 원칙
- 작업 진행 중 상태가 바뀌면 관련 문서를 즉시 갱신한다
- 최신 상태는 `docs/status.md`에 유지한다
- 중요한 의사결정 변경은 `docs/decisions.md`에 누적 기록한다
- 실제 작업 진행 내역은 `docs/logbook.md`에 시간순으로 남긴다
- 코드, 데이터, 운영 기준이 바뀌었는데 문서가 뒤처지지 않도록 항상 함께 관리한다

### 3-5-3. 환경 문서 갱신 원칙
- Linux 서버 학습 환경은 `docs/envs/wake_word_env.md`, Jetson runtime 환경은 `docs/envs/jetson_wake_word_env.md`를 기준 문서로 사용한다
- Python 환경, venv 경로, conda 환경명, JetPack/L4T, CUDA/TensorRT, ORT 버전 또는 설치 위치가 바뀌면 같은 세션에서 해당 env 문서를 바로 갱신한다
- 환경 세팅 문서에는 목적, 경로, 설치 절차, 검증 명령, 현재 검증 결과를 함께 유지한다
- 새 세션에서 바로 따라갈 수 있게, 문서 허브(`docs/README.md`)에도 관련 env 문서를 노출한다

### 3-5-1. Git 연동 시 버전 관리 원칙
- Git으로 작업을 관리하는 경우, 코드 변경과 문서 변경을 가능한 한 같은 흐름 안에서 함께 정리한다
- 기능 추가, 데이터 처리 기준 변경, 학습 방식 변경, 디렉토리 구조 변경이 발생하면 관련 문서와 Git 상태를 같이 점검한다
- 중간 산출물과 재생성 가능한 대용량 파일은 버전 관리 대상에서 분리하고, 추적 대상은 의도적으로 선택한다
- 다만 Jetson runtime 재현에 직접 필요하고 용량이 작은 ONNX 자산은 예외적으로 추적할 수 있다
- 커밋 전에는 현재 변경 사항이 문서 기준과 일치하는지 확인하고, 추적되면 안 되는 파일이 포함되지 않았는지 점검한다
- 작업 단위가 정리됐다고 판단되면 중간 커밋은 수시로 수행한다
- `secrets/` 아래 파일은 어떤 경우에도 Git 추적, 커밋, 원격 반영 대상에 올리지 않는다
- 커밋과 원격 반영 전에는 `secrets/`가 staged 또는 tracked 상태가 아닌지 반드시 확인한다
- `secrets/` 또는 민감한 운영 메모가 추적 상태로 보이면 커밋/푸시를 멈추고 먼저 추적 해제 또는 로컬 전용 문서로 이동한다

### 3-5-2. 민감 정보 문서화 원칙
- 내부 주소, 계정 식별자, SSH 경로, 비공개 운영 메모는 공개 문서에 직접 남기지 않는다
- 이런 정보가 꼭 필요하면 `secrets/` 아래의 로컬 전용 문서로 분리해 관리한다
- 공개 리포에는 일반화된 설명만 남기고, 실제 접속 정보나 운영 절차는 로컬 메모로 분리한다
- 문서 정리 과정에서 민감 정보가 발견되면 최신 공개 문서에서는 제거하고, 필요한 경우에만 로컬 전용 문서로 이동한다
- 원격 반영 방식과 무관하게 위 원칙은 예외 없이 유지한다

### 3-6. 대용량 데이터 처리 원칙
- 대용량 다운로드 파일은 손상 여부가 의심되어도 즉시 삭제하지 않는다
- 먼저 파일 헤더, 압축 무결성, 이어받기 가능 여부를 확인한다
- 이미 일부라도 다운로드된 파일은 가능하면 보존하고 `resume` 또는 검증을 우선 시도한다
- 삭제가 필요할 때는 진행 손실과 대체 경로를 함께 검토한 뒤 수행한다
- 큰 압축 병합/해제 작업은 가능한 한 로그 파일 또는 조용한 모드로 실행한다

### 3-7. 장시간 실행 로그 원칙
- 1분 이상 걸릴 수 있는 모든 실행 코드는 기본적으로 주기 로그를 출력한다
- 기본 주기는 1분이며, 진행률, 처리량, 현재 단계, 예상 남은 시간을 가능한 범위에서 함께 출력한다
- 사용자가 별도로 물어보지 않아도 상태를 확인할 수 있도록 터미널 출력만으로 진행 상황을 파악 가능해야 한다
- 인터럽트 이후 재실행이 잦은 작업은 부분 재실행 옵션과 함께 설계하는 것을 우선한다

## 5. Wake Word 개발 방침

### Wake Word: "하이 포포"

### 학습 전략
- **모델**: openWakeWord (Apache 2.0)
- **Backbone**: Google Speech Embedding (Apache 2.0, frozen ONNX)
- **분류기**: FC 2레이어(32유닛) → binary cross-entropy, PyTorch 학습 → ONNX export
- **학습 환경**: Linux PC (A100 80GB, CUDA 11.8, conda 환경 `wake_word`)
- **추론 환경**: Jetson (onnxruntime-gpu)
- **참고 노트북**: openWakeWord 레포 `notebooks/training_models.ipynb` (한국어 미지원이라 커스텀 파이프라인 작성)
- **실행 의존성**: 현재는 openWakeWord 로컬 clone 없이, 리포 안의 `features.py`와 `assets/feature_models/*.onnx`만 사용

현재 구현 상태 메모:

- classifier checkpoint 학습과 평가까지 완료
- classifier ONNX export 완료
- `detector.py`는 classifier ONNX 래퍼와 raw audio 실시간 추론 래퍼를 함께 제공한다
- feature 추출은 `melspectrogram.onnx`와 `embedding_model.onnx`를 ONNX Runtime으로 실행한다
- `wake_word_gui_demo.py`로 Jetson 기본 마이크 실시간 검증이 가능하다

### VAD 전략

- **기본 backend**: Silero VAD ONNX
- **비교 backend**: webrtcvad
- **공통 진입점**: `vad/detector.py`
- **공통 인터페이스**: `infer(audio_chunk) -> bool`
- **기본 filtering**:
  - `min_speech_frames=3`
  - `min_silence_frames=10`
- **실행 검증**: `vad/vad_demo.py`로 기본 마이크 기준 `status / level / conf` 확인
- **현재 단계**: 요소기술 완료, 다음 단계는 wake word 뒤 연동과 utterance segmentation 기준 정리

### 데이터 전략
- **Positive 목표**: 1,000~2,000개
  - Edge TTS 한국어 10개 목소리 × 속도/피치 변형
  - 실제 녹음: 팀원 8명 × 12~15회 (총 ~100개)
  - librosa 증강으로 3~5배 확장
- **Negative**: AI Hub 자유대화 음성(일반남여) + MUSAN + FSD50K
  - 한국어 일상 발화는 AI Hub를 우선 사용
  - 환경음/잡음 다양성은 MUSAN, FSD50K로 보강
  - 세부 수집 기준은 `docs/research/negative_datasets.md` 참고

### TTS 전략 (Positive 샘플 생성)
| 도구 | 라이선스 | 용도 |
|------|---------|------|
| Edge TTS | Microsoft 약관 | PoC 데이터 생성 (10개 목소리, 속도/피치 조절) |
| MeloTTS | MIT | 제품화 시 대체 (오프라인, 라이선스 안전) |

## 6. 개발 프로세스 (PDCA)

```
Plan  → 기술 조사, 모델 선정, 데이터 전략 수립
Do    → 학습 파이프라인 구현, 모델 학습, 모듈 코드 작성
Check → 데모 검증, 성능 지표 측정 (FAR/FRR), 엣지케이스 확인
Act   → 임계값 튜닝, 데이터 보강, 모델 교체, 다음 사이클 반복
```

각 사이클의 결과는 `docs/status.md`, `docs/decisions.md`, `docs/logbook.md`에 함께 기록한다.
